## ‚úÖ Selected Candidates (Nav: Selected Candidates)
Purpose: send selection emails and view status
- Endpoint:
  - POST `/selections/notify` (protected)
    - Body: `{ subject?, message?, applicationIdList: ["id1","id2", ...] }`
    - Response: `{ success: true, results: { sent: [...], failed: [...] } }`
- UI:
  - Selected candidates page: table filtered to `status === SELECTED` or allow selecting rows and pressing ‚ÄúSend Selection Email‚Äù
  - Batch UI: checkbox select multiple, open ‚ÄúSend Selection Email‚Äù modal with Subject & Message, Confirm -> API call
  - Show per-application result: success list / failed list with reasons
- UX notes:
  - Disable button while in progress; show spinner
  - Show counts: Sent X / Failed Y and per-row error messages
  - Mark application status to `SELECTED` on success (backend already updates)

## üóì Interview Schedule (Nav: Interview Schedule)
Purpose: schedule interviews and notify shortlisted applicants
- Endpoint:
  - POST `/interviews/:applicationId/schedule` (protected)
    - Body: `{ interviewDate, interviewTime, mode, interviewLink, interviewer, notes }`
    - Response: success message
- Behavior:
  - On save, backend triggers emails to shortlisted applicants and updates application status to `INTERVIEW_SCHEDULED` (controller handles send + status update).
- UI:
  - Schedule page: Form to pick application (searchable), date/time pickers, mode (Online/In-person), link, interviewer, notes
  - Schedule history: list scheduled interviews per application/job
  - Notifications: confirmation when emails successfully queued/sent; show failures
- UX:
  - Calendar view (optional) with scheduled interviews by date
  - Edit/cancel schedule (if implemented server-side) ‚Äî show disabled if not supported
